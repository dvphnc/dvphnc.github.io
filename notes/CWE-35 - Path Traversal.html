Path Traversal

Directory Traversal — When attackers walk outside the web root

Path Traversal allows attackers to access files and directories outside the intended web root by manipulating file path parameters — often using ../ sequences.

How It Works

The application uses unvalidated user input directly in filesystem paths → attacker crafts ../../etc/passwd → server happily serves restricted files.

Deadly Consequences

  • Read /etc/passwd, config files, source code
  • Steal database credentials
  • Enable further compromise & privilege escalation
  • If write access exists → full server takeover

Traversal Techniques (Bypass Arsenal)

  • ../../etc/passwd
  • %2e%2e%2f%2e%2e%2fetc%2fpasswd (URL encoding)
  • %252e%252e%252f (double encoding)
  • ....//....//etc/passwd (filter evasion)
  • ..%00/ (null byte — legacy systems)

Prevention — Lock It Down

  • Never trust user input in file paths
  • Use allowlists (e.g., map IDs → real files)
  • Normalize + validate: resolve .. then check if path starts with base directory
  • Run app with least privilege
  • Chroot / containerize if possible
  • Decode input before validation

Testing Tools

  • Burp Suite (Intruder + payloads)
  • OWASP ZAP
  • DotDotPwn (dedicated traversal fuzzer)
  • DirBuster / ffuf

Final Word

Path Traversal isn’t loud. It doesn’t crash servers.
It just quietly walks past your defenses and reads everything you tried to hide.

Validate. Normalize. Contain.
Or the filesystem becomes public.
